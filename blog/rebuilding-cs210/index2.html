Rebuilding CS210: Bridging Python and Data Structures with C

Notes from redesigning the C programming course that sits between an introductory Python course and a systems-heavy data structures course. The bridge is the whole point, and most of the design decisions come down to figuring out what to load-bear and what to defer.

CS210 is the middle child of our intro CS sequence. Students come in from CS110 having written a semester of Python: they know how to read a loop, call a function, and reason about a list. They leave heading into CS220, where they will implement linked lists, hash tables, and binary trees in C and argue about asymptotic complexity. CS210 has to take them from one to the other, and it has fifteen weeks to do it.

I took over as Course Director knowing the course needed a real rebuild rather than incremental cleanup. The previous version worked, but the seams were showing in predictable places: students could complete the C labs without ever building a mental model of memory, the toolchain story was muddled, and the lab autograding was duct-taped together in a way that made it hard to give cadets meaningful feedback. This post is a tour through the design decisions I'm making now, before the fall offering, and the tradeoffs each one carries.

What the bridge is actually carrying

Before redesigning anything, I forced myself to write out, in plain prose, what CS210 has to deliver so that CS220 can start where it expects to start. The answer turned out shorter than I expected:

  1. Read and write C confidently at the level of a small program with several functions, a few structs, and a main loop.
  2. Reason about memory: stack vs. heap, when something is allocated, when it is freed, what happens when you forget.
  3. Use pointers fluently for the cases that matter — passing things by reference, walking arrays, holding the address of a struct.
  4. Compile, link, and run a program from a terminal. Read a compiler error. Read a runtime error. Use a debugger when reading isn't enough.
  5. Internalize that the language is not the IDE, and the program is not the source file.

Everything else — file I/O patterns, bitwise tricks, makefiles, multi-file projects, fancy preprocessor work — is on a "nice to have if there's time" list. The course has fifteen weeks and thirty-five labs. The list above is what has to survive.

The hard part of CS210 isn't C. It's the gap between "I wrote a Python script that runs" and "I have a model of what the machine is doing." Cadets who treat C as Python with semicolons get punished by segfaults they can't reason about. The course has to make that transition explicit.

Toolchain decisions, deferred carefully

One of the more interesting calls is which environment cadets will use. The current setup is VSCode plus WSL plus gcc — workable, but with rough edges. WSL drives some students to distraction with file-system quirks, and the VSCode-as-everything model can mask the fact that compilation is a discrete step rather than a "Run" button. We are likely moving to a Linux VM in the near term, which fixes some problems and introduces others.

Rather than commit course materials to a specific toolchain that may not survive the year, I am writing the lessons and labs to be toolchain-agnostic wherever possible. References say "your terminal" and "your gcc" rather than naming a specific IDE. The first-lesson setup checklist is the one piece that will need a real rewrite when the decision is final, but everything downstream stays portable.

This is more painful in the short term — you cannot show a cadet exactly where to click — and easier in the long term, because the materials survive the next toolchain conversation, which is always coming.

Lab autograding without GitHub Actions

For Lab 02 (an altitude and temperature calculator, the first one with real arithmetic and real structure), I built out the autograding bundle and converged on an approach that's worth describing because it differs from what most CS courses do.

The standard pattern is to push student code to GitHub, trigger a GitHub Actions workflow, run the tests, and surface results. It works, but Actions adds a layer of CI/CD complexity that cadets don't need at this stage of their development. It also makes it harder to give cadets a self-test they can run locally before submitting, which is the whole point of giving them tests in the first place.

Instead, I'm using inline assert statements guarded by a preprocessor flag: 

#ifdef SELFTEST
    assert(altitude_to_pressure(0) == 1013);
    assert(altitude_to_pressure(5000) < 540);
#endif

The cadet writes their function. They can compile with -DSELFTEST and run the assertions before they submit. Gradescope compiles with the same flag on the back end and grades against the same checks the cadet has been able to see all along. There is no surprise grading rubric. If their self-test passes, the autograder will agree.

This has a pedagogical benefit beyond convenience: it normalizes the idea that tests are part of the program, not a separate thing the instructor does to you. By the end of CS210, cadets have been reading and writing assertions for fourteen weeks. When they hit CS220 and start writing unit tests for their own data structures, the muscle is already built.

The Python-to-C culture shock, and what to do about it

The single biggest predictor of struggle in CS210 isn't whether a cadet did well in CS110. It's whether they are willing to slow down enough to think about what the machine is doing. Python hides almost everything that matters in C. Cadets who treated Python as a magic incantation that sometimes works can no longer do that, because the magic now segfaults.

I'm leaning hard on three techniques to make this transition explicit rather than implicit:

Diagrams before code

When introducing pointers, the first half of the lesson is hand-drawn boxes and arrows. Variable on the stack, value inside. Pointer is a different box, value is the address of the first box. Dereference follows the arrow. Only after the diagrams do we write the code. Cadets who see *p = 5 before they have a picture of what p points to are memorizing syntax. Cadets who have drawn it first are translating from a picture.

Compile-error reading as a graded skill

Reading a C compiler error is its own competency, separate from writing C. I'm building it into the rubric for the first several labs: not just "does it compile and pass tests," but "given this broken code, what does the error mean and where is the bug?" Cadets who can read an error message are independent. Cadets who can't are dependent on whichever instructor is on EI.

Frequent, low-stakes retrieval

This one I've written about elsewhere and is the subject of a SoTL project I presented as a poster at the USAFA SoTL conference earlier this year. The short version: spaced, repeated, low-stakes testing on small concepts is one of the most reliably evidence-supported techniques in CS pedagogy, and it is dramatically underused in our courses. CS210 will use it heavily in the form of two-minute warm-up questions at the start of every lesson. We will see whether the effect I measured in CS110 replicates in C, which is its own interesting question.

Things I'm explicitly not doing

A course redesign is mostly a series of decisions about what to leave out. A few items I considered and then decided to defer:

  • Makefiles. They are useful, they will be useful in CS220, and they are an unnecessary cognitive tax in week three of a course that is already asking a lot. We will compile with explicit gcc commands all semester. Makefiles can wait for CS220.
  • Multi-file projects beyond the bare minimum. One .c file and the standard library covers most labs. We do introduce header files and separate compilation toward the end, but the early labs are deliberately small and contained.
  • Bitwise operators as a topic of their own. They appear in context where natural (mask flags, simple checksums) but do not get a dedicated lesson. CS220 and CS380 have better homes for them.
  • Heavy use of GenAI tooling. Cadets are aware of it and use it on their own. The course will not lean on it for explanation or scaffolding, because the failure mode — cadets who can produce code without being able to read or debug it — is exactly the failure mode CS210 is supposed to prevent. I think there is interesting work to do on integrating these tools at higher levels, and I am writing about it separately, but CS210 is not the place.

What I am watching for

A redesign isn't done when the lessons are written. It is done when you have iterated on the version that actually meets students in the room. A few things I will be measuring this fall and adjusting in the spring:

  • Lab completion rates, especially for the first three pointer-heavy labs.
  • EI request volume and topic clustering — what cadets are coming in for is a fast read on what the lessons did not land.
  • CS220 instructors' impressions in the spring — the bridge has a downstream consumer, and they will tell me if the cadets are arriving with the prerequisites or without them.
  • Retrieval-practice warm-up performance over the semester, both as a learning signal and as data for the SoTL replication.

None of this is groundbreaking pedagogy. The pieces — diagrams first, error reading as a graded skill, self-tests baked into labs, retrieval practice as a warm-up — are well-supported in the CS education literature. The work is in committing to a coherent set of them, building the materials consistently around that commitment, and being willing to revise when the next cohort tells you something you weren't expecting.

I'll write a follow-up after the fall semester, with what worked, what didn't, and what I'd change for the second offering. If you're rebuilding a similar course and want to compare notes, I'd genuinely like to hear what you're trying.
